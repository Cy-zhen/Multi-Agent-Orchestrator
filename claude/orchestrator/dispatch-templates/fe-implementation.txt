## 编码原则

1. 组件拆分到 Radix UI 可直接复用
2. 页面级加载/空/错误状态必须覆盖
3. 响应式至少 3 断点（375/768/1200）
4. API 层统一用 Axios + interceptor
5. 如果项目已存在页面/组件/路由，优先在现有实现上重构，不要只平行新增一套新页面
6. 如果 PRD 提的是“重构”，必须明确说明你重构了哪些旧页面/逻辑，而不是只补了新代码
7. 编码完成后运行 `yarn build` 验证

## 🛑 Red Flags 自检（被动技能: Superpowers/Discipline）

| 你在想什么 | 实际应该做什么 |
|-----------|-------------|
| "跳过 build 验证直接输出" | build 不可跳过 |
| "这个边界不会出现" | 写 loading/empty/error 状态，让证据说话 |
| "先核心功能再错误处理" | 错误处理是核心功能的一部分 |
| "我知道代码是对的" | 跑 `yarn build`，让编译器证明 |
| "顺便改个不相关的东西" | 不要。只做 PRD 范围内的事 |
| "差不多就行了" | 完整实现或不做 |

`Agent Updates` 追加格式如下：

```md
#### {UTC 时间} | FE | build
- Completed: 本次完成的页面/组件/API 对接
- Decisions: 关键实现决策、约束、妥协
- Files: 修改过的核心文件
- Blockers: 当前阻塞；没有则写 `无`
- Handoff: 需要 BE / QA / 下一个 FE 会话继续处理的点
- Next: 如果会话现在中断，下一个 Agent 最应该先做什么
```
