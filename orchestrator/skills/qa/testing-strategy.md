# 测试策略

## 核心规则

### 测试金字塔
```
        /  E2E  \          ← 少量（关键用户流程）
       / 集成测试 \         ← 中量（API + 数据库交互）
      /  单元测试   \       ← 大量（纯逻辑函数）
```

### 用例设计原则
1. **先读 PRD 验收标准**：每条验收标准至少 1 个测试用例
2. **等价类划分**：正常值、边界值、异常值各一组
3. **状态转换覆盖**：有状态的功能必须覆盖所有状态转换路径
4. **错误路径优先**：先写失败用例，再写成功用例

### 覆盖率目标
- 核心业务逻辑：≥ 80%
- API handler：≥ 60%
- UI 组件：≥ 50%（重交互的 ≥ 70%）
- 工具函数：≥ 90%

### 测试命名
```
test_[功能]_[场景]_[预期结果]

示例：
test_login_with_invalid_password_returns_401
test_transfer_exceeding_balance_throws_insufficient_funds
```

### 禁止
- ❌ 测试之间有依赖（每个测试独立可运行）
- ❌ 测试依赖外部服务（用 mock/stub）
- ❌ 只测试 happy path（必须覆盖错误路径）
- ❌ 硬编码测试数据（用 fixture/factory）
